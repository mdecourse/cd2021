分組專案設計與簡報重點:

  1. 在無實體原型製造情況下, 是否能夠產出自我驗證的結果?
  2. 設計流程若能輔以理論分析或文獻參考與探討佐證, 將可凸顯題目或結果的重要性.
  3. 善用 What? Why? When? Who? How? and Where? 式的自我定性與定量檢視, 將各種資源用於刀口並呈現歷程.
  4. 簡報若能以 Web based 投影片為主, 可方便使用者隨時參閱, 簡報內容要呈現重點, 而且文句經過仔細查驗與校正.

協同產品設計工具

CoppeliaSim (之前稱為 Virtual Robot Experimentation Platform, V-REP)

從 CoppeliaSim 之前的名稱, V-rep 比較能夠了解這項工具的特性, 也就是所謂的虛擬機器人實驗平台.

V-REP Lecture.pdf (V-rep 功能簡介)

將具備 Kinematic, Dynamic 與 Automatic Control 特性的產品設計從 MCAD 設計繪圖套件將零組件轉入 CoppeliaSim 平台進行虛擬實驗的目的, 就是希望展示其:

  1. 運動特性 - Kinematic Characteristics, https://en.wikipedia.org/wiki/Kinematics
  2. 動力特性 - Dynamic Characteristics, https://en.wikipedia.org/wiki/Dynamics_(mechanics)
  3. 可受控制特性 - https://en.wikipedia.org/wiki/Controllability

提供 Proximity Sensor (鄰近感測器), 可以感測並回授鄰近物體的訊號, 通常用於行進間避開障礙物或計算通過物數量.

提供 Vision Sensor (影像感測器), 可以用於影像辨識 (回授導引行進) 或監控 (串流).

提供 Calculation Module (運算模組), 可以進行各種理論數值分析運算 (避障, 干涉, 位移, 速度, 加速度).

而且設計者可以利用 Lua Scripting 以及 Python Remote API 的方式直接或遙控的方式進行互動控制或監測. 使用 CoppeliaSim 進行產品系統的虛擬展示目的, 是作為後續原型實作測試之前的重要評量與設計內容取捨用.

Programming_in_Lua.pdf

這個禮拜開始將要著手研究 https://www.coppeliarobotics.com/, 從下載教育版的檔案中, 可以發現最新的版本為 4.1.0, 於 2020.07.21 推出.

Coppeliasim 的 documentation 位於 helpFiles 中的 en 目錄中, 且說明檔案中的影像檔案位於 images 中. 假如將 en 目錄中的所有檔案分兩次 (避免一次提交推送超過 1000 個檔案), 送到 https://github.com/mdecourse/copsim 倉儲中的 content 目錄, 且將 main 分支設為 Github Pages 的根目錄, 可以得到:

https://mde.tw/copsim

由於 https://mde.tw/cmstemplate 的資料架構, 其靜態網站位於 content 目錄中, 且索引檔案名稱為 index.html, 因此前述的 content 檔案複製中, 已經將 en 目錄中的導引檔案 welcome.htm 複製一份命名為 index.html.

但如此設置的 https://mde.tw/copsim 無關鍵字查詢的功能, 也沒有動態網頁的內容, 比較不容易修改註記, 所以接下來要研究該如何處理此一架構的文檔, 讓它成為協同產品設計實習流程在研究 Coppeliasim 時的靈活參考文件.

而 github.com 的免費倉儲使用有總檔案數與個別檔案容量大小的限制, 因此最後所完成的 copsim 倉儲或許要以工作站中的主機採 https://fossil.kmol.info 的方式伺服.

Webots

研究 https://cyberbotics.com/ 也必須從其技術手冊下手. Webots 在 2018年12月18日宣布開源, Webots 的公司訊息以 twitter 發布. Webots 的使用對商業公司採取比 CoppeliaSim 更加友善的授權. 主要的差別是: Webots 套件不僅開源, 而且允許營利或教育單位免付費情況下使用, 也就是必須在遵循 https://www.apache.org/licenses/LICENSE-2.0 授權下使用, 但 CoppeliaSim 雖然也是開源, 但只允許教育單位免費使用, 營利或商業使用則必須付費取得授權.

Onshape and A360

從協同產品設計的角度, 除了善用 Github, Gitlab, Heroku 與 Fossil SCM 外, 用於零組件設計繪圖的工具也應優先採用 cloud based 的 Onshape 與 A360. 主要原因除了避免各參與協同設計的組員採用不同版次的 MCAD (Mechanical Computer Aided Design) 套件產生問題外, 也希望能夠將平板與手機導入機械設計工程師的產品開發流程中.

Onshape 的基本介面程式採用 https://cad.onshape.com/FsDoc/
